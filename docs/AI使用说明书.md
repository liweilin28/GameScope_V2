# AI 使用说明书

## 一、说明目的

本说明书用于透明记录本项目在开发过程中对 AI 工具的使用方式、使用边界、人工修改与验证过程，满足课程对“AI 使用透明度”和“代码来源说明”的要求。

## 二、总体原则

1. AI 是辅助工具，不替代学生完成选题、功能取舍和最终提交判断。
2. 所有关键计算结果必须来自 Python 后端，不允许由 LLM 直接编造统计结论。
3. AI 生成的代码、文档和提示词必须经过学生人工阅读、修改、本地运行和测试验证后才可保留。
4. 当 AI 建议与课程要求、项目稳定性或数据真实性冲突时，以课程要求和人工验证结果为准。

## 三、本项目使用到的 AI 工具

- 对话式大语言模型：用于需求拆解、代码草稿、调试建议、文档初稿
- 可选 LLM 接口：用于创意解析、自然语言解释和报告润色

说明：项目运行时的可选 LLM 仅在学生自行配置环境变量后启用；未配置时系统自动使用规则 fallback。

## 四、关键使用案例

### 案例 1：项目架构重整

- 任务目标：将原先偏原型化的思路整理为可提交的 FastAPI + pandas + HTML/CSS/Vanilla JS + ECharts 项目结构
- 使用工具：对话式大语言模型
- Prompt 摘要或关键片段：要求保留 Python 数据分析后端，不使用 React、Vue、Streamlit 和 npm 构建流程，并按课程五大模块组织页面与接口
- LLM 返回内容摘要：给出后端 API、前端页面、数据服务层和测试文件的模块划分建议
- 学生如何修改 / 验证：学生保留现有技术栈，删除不符合课程约束的替代方案，结合本地目录逐步调整
- 最终结果：形成当前项目的前后端分层结构与模块命名方式
- 是否采纳：采纳，但由学生按课程要求做了结构收敛和命名修正

### 案例 2：数据清洗与字段兼容

- 任务目标：实现默认 Steam CSV 与用户上传 CSV 的兼容清洗
- 使用工具：对话式大语言模型
- Prompt 摘要或关键片段：要求兼容 `Name`、`Release date`、`Positive`、`Negative` 等字段名差异，并生成评论数、好评率和年份字段
- LLM 返回内容摘要：生成字段标准化、数值转换、缺失值处理和衍生字段函数草稿
- 学生如何修改 / 验证：学生补充了去重逻辑、无 `name` 时的通用 fallback、警告信息和 pytest 测试
- 最终结果：上传字段不完整或字段命名不统一的 CSV 时，系统能尽量读取并给出清晰提示
- 是否采纳：部分采纳，核心逻辑保留，异常处理和测试由学生补强

### 案例 3：Dashboard 图表与数据发现

- 任务目标：让市场总览页面具备课程展示所需的指标卡、趋势图和动态分析结论
- 使用工具：对话式大语言模型
- Prompt 摘要或关键片段：要求使用 ECharts 展示发行趋势、Genre/Tag 分布、价格和好评率分布，并补一个紧凑的分析结论区域
- LLM 返回内容摘要：给出前端指标卡布局、图表调用组织方式和简要的结论区域思路
- 学生如何修改 / 验证：学生将“结论”严格改为 pandas 统计结果生成，不允许 LLM 直接产出统计发现；本地检查图表空状态与真实数据一致性
- 最终结果：Dashboard 页面可同时展示“图表结果”和“数据发现”短结论，便于课堂讲解
- 是否采纳：采纳，但统计结论的生成方式由学生重新约束

### 案例 4：Data Q&A 规则问数

- 任务目标：实现基础自然语言问数能力，并确保回答来源于真实计算
- 使用工具：对话式大语言模型
- Prompt 摘要或关键片段：要求先做问题分类，再调用后端分析函数，最后由模板或 LLM 解释结果
- LLM 返回内容摘要：给出问题分类、问答模板和接口结构建议
- 学生如何修改 / 验证：学生限制了 AI 的角色，只允许解释和润色，不允许直接产生“统计答案”
- 最终结果：Data Q&A 可进行交互式问答，同时保留课程要求中的真实性和可解释性
- 是否采纳：采纳

### 案例 5：Idea Lab PLUS 创新

- 任务目标：为课程加分项设计一个能讲清楚创新价值的模块
- 使用工具：对话式大语言模型
- Prompt 摘要或关键片段：要求围绕“独立游戏立项辅助”生成创意解析、竞品雷达、机会评分和差异化建议链路
- LLM 返回内容摘要：给出创意画像字段、相似竞品说明格式和 Project Brief 组织方式
- 学生如何修改 / 验证：学生将机会评分固定为规则与统计驱动，不让 LLM 参与分数计算；并补充支持数据和边界说明
- 最终结果：Idea Lab 成为项目的 PLUS 创新功能，可在课堂演示中单独说明创新点
- 是否采纳：采纳，但评分机制和边界提示由学生人工重构

### 案例 6：README 与提交文档整理

- 任务目标：将 README、测试记录、AI 使用说明和提交清单整理成可提交材料
- 使用工具：对话式大语言模型
- Prompt 摘要或关键片段：要求按课程 PPT 的验收逻辑组织文档，并加入运行、测试和演示路径
- LLM 返回内容摘要：给出 README 章节提纲、课程验收表和文档模板
- 学生如何修改 / 验证：学生将其中不准确或未验证的表达删除，补上真实测试命令、样例数据规模和待补充事项
- 最终结果：文档更接近课程提交物，而不是普通代码仓库说明
- 是否采纳：采纳

## 五、未采纳 AI 建议示例

### 示例：使用前端框架重写页面

- AI 建议摘要：建议使用 React 或 Vue 统一组件化管理页面状态
- 未采纳原因：课程要求和当前项目约束明确不更换技术栈，也不引入 npm 构建流程
- 学生决策：继续保持 HTML/CSS/Vanilla JS 架构，只在原有页面结构内做增量增强

### 示例：让 LLM 直接生成市场分析结论

- AI 建议摘要：直接根据问题生成“市场趋势总结”自然语言段落
- 未采纳原因：这类方案容易脱离真实数据，违反“LLM 不做统计计算”的项目边界
- 学生决策：改为由 pandas 先计算，再由模板或可选 LLM 做解释

## 六、学生关键决策清单

- 确定主题不是做游戏，而是做游戏市场分析平台
- 保留 FastAPI + pandas + HTML/CSS/Vanilla JS + ECharts 技术栈
- 不使用 React、Vue、Streamlit 和 npm 构建流程
- 将 LLM 功能限制在解析、解释和润色，不参与真实统计计算
- 将 Idea Lab 设定为 PLUS 创新功能
- 增加课程验收面板与 submission readiness 接口，强化课堂可见性
- 增加无 `name` 通用 CSV fallback，提升上传数据鲁棒性
- 用 pytest 持续验证关键数据清洗和分析逻辑

## 七、人工验证方法

- 本地启动后端并手动打开前端页面检查交互
- 查看 Data Pipeline 页面是否能正确显示原始预览、清洗预览和清洗报告
- 检查 Dashboard 图表是否与当前数据一致
- 使用 pytest 回归关键模块
- 检查未配置 LLM 环境变量时系统是否稳定回退到规则模式

## 八、AI 使用边界总结

- AI 可以提升开发速度，但不能替代课程项目中的真实性、稳定性和最终责任
- AI 输出内容必须通过人工理解和验证后再进入提交版本
- 对于数据结论、课程验收口径和来源说明，必须以人工确认结果为准

## 九、反思总结

AI 在本项目中最有价值的地方，是帮助快速生成代码骨架、接口初稿、测试样例和文档框架，减少重复劳动。

AI 最容易带来的风险，是为了“回答完整”而给出未经验证的实现细节或看似合理但并非真实统计结果的分析语言。因此，学生在本项目中采取的策略是：让 AI 参与生成候选方案，但把数据真实性、功能边界、技术栈约束和最终提交判断牢牢保留在人类决策层。

如果后续继续迭代，本项目更适合在固定数据字典、明确课程边界和已有测试约束下继续使用 AI，而不是让 AI 自由扩展功能范围。
